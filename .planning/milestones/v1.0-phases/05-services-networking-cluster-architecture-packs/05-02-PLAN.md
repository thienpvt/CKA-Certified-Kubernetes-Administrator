<parallel_safety>
Wave 2 merge discipline (applies to ALL plans in this phase that append to pack-level files — manifest.yaml, coverage.yaml, README.md):

The append pattern uses sentinel-bracketed blocks (`# BEGIN phase-05 new questions` / `# END phase-05 new questions`) plus `grep -q <question-id> <file> || sed -i ...i...` idempotency guards. This makes each `sed -i` rerun-safe in isolation but does NOT make concurrent commits to the same base branch race-free.

Serialization rule: within a single pack (services-networking OR cluster-architecture), Wave 2 plans merge back to the base branch ONE AT A TIME. The executor runs plans in parallel against separate worktrees, but the final merge-back (or direct push to the base branch) is serialized — next plan rebases on the prior plan's merge before pushing. This avoids the three-way-merge ambiguity that arises when two plans append adjacent lines inside the same sentinel block in the same commit range.

Cross-pack (S&N plans merging while Cluster-Arch plans merge) is safe — different files, no overlap.

Execute-phase honors this by sequencing Wave 2 merge operations per pack. Wave 1 (independent-file work: individual question dirs, fixtures, lib helpers) remains fully parallel.
</parallel_safety>
