## Baseline Capture Mechanism

### Q1: Where should the post-setup baseline live, and who writes it?

| Option | Description | Selected |
|--------|-------------|----------|
| Runner-managed, /tmp file | Runner calls capture AFTER setup returns; writes `/tmp/cka-sim/<question>/baseline.json`. Author-discipline-free. | ✓ |
| Setup-managed, /tmp file | Each setup.sh calls a helper at its end. Author must remember. | |
| Lab-namespace annotation | Baseline JSON as `cka-sim.io/baseline` annotation on lab ns. Inside cluster; large jq escaping. | |
| Declarative baseline.yaml | Static per-question YAML listing expected post-setup state. Drift risk vs setup. | |

**User's choice:** Runner-managed, /tmp file
**Notes:** Removes "easy to forget" failure mode — runner enforces contract once for every question.

### Q2: What fields should the runner capture per resource at baseline time?

| Option | Description | Selected |
|--------|-------------|----------|
| Lean — generation, resourceVersion, kind+name+namespace, labels | Smallest baseline supporting all 3 delta helpers + ownership. ~1–5 KB. | ✓ |
| Lean + content hash | Adds SHA256 of .spec. Catches label-only / annotation-only changes that don't bump generation. | |
| Full kubectl snapshot | Whole `kubectl get -o json`. Most flexible, largest payload, slowest. | |
| Custom per-question fields | Author declares which fields to baseline per question. Author-discipline-heavy. | |

**User's choice:** Lean — generation, resourceVersion, kind+name+namespace, labels
**Notes:** Plus flat resource-list (kind/name) for authorship checks. Content hash deferred to v1.x if a future question needs it.

## Delta-Helper + Trap-Ownership API

### Q1: How should `assert_changed_since_setup <kind> <name>` decide a resource was modified?

| Option | Description | Selected |
|--------|-------------|----------|
| Generation OR resourceVersion bump | Generation catches spec edits; resourceVersion catches any kubectl apply. Cheap, robust. | ✓ |
| Generation only | Misses label/annotation-only changes. | |
| Spec-hash diff | SHA256 of .spec compared to baseline hash. Requires content hash in baseline. | |
| Flag-driven (caller picks) | `--by gen|rv|spec` per assertion. Pushes choice onto authors. | |

**User's choice:** Generation OR resourceVersion bump

### Q2: How should `assert_resource_candidate_authored <kind> <name>` decide a resource was created by the candidate?

| Option | Description | Selected |
|--------|-------------|----------|
| Not in baseline resource-list | Pass iff kind/name absent from baseline.resource_list[]. Simple + deterministic. | ✓ |
| Annotation marker | Setup labels resources with `cka-sim/owner=setup`. Helper checks absence. Retrofit every setup. | |
| creationTimestamp > baseline_capture_ts | Timestamp resolution is seconds; rare corner case. | |
| UID not in baseline UID-set | Catches delete+recreate as authorship. | |

**User's choice:** Not in baseline resource-list
**Notes:** Sufficient for the 4 known offenders. UID-based check deferred to v1.x if delete+recreate false positives surface.

### Q3: How should `lib/traps.sh` detectors verify ownership before firing?

| Option | Description | Selected |
|--------|-------------|----------|
| Detector signature requires baseline-aware check | Refactor existing detect_<id>() to take subject + check baseline. Same baseline data. | ✓ |
| Wrapper helper | `guard_ownership <kind> <name> -- detect_<id>` at call site. | |
| Catalog-driven | traps/catalog.yaml gains `ownership: required|optional|always` field; runtime honors automatically. | |
| Per-detector author choice | No structural requirement; per-detector docs. | |

**User's choice:** Detector signature requires baseline-aware check
**Notes:** Catalog gets `ownership:` tag (candidate-required default, setup-allowed for content-bug traps) so author-intentional traps like `pss-error-string-mismatch` still fire.

## CI Regression Strategy

### Q1: Where should the 'empty submission scores 0' check run, and against what?

| Option | Description | Selected |
|--------|-------------|----------|
| Mocked kubectl unit test in test.sh | Per-question test under tests/grading-honesty/, PATH-shadowed kubectl. Seconds, runs on every PR. | ✓ |
| Real-cluster smoke in separate GHA job | kind or GKE per PR. 5–20 min; needs credentials in CI. | |
| Hybrid — mocked + optional real smoke | Mocked always, real on nightly / [smoke] label. More CI infra. | |
| Local-only `lint-grading-honesty.sh` | Static linter inspecting grade.sh patterns. Not behavioural. | |

**User's choice:** Mocked kubectl unit test in test.sh

### Q2: What's the per-question test fixture shape?

| Option | Description | Selected |
|--------|-------------|----------|
| Capture-then-replay | One-time live-cluster setup → snapshot kubectl outputs → commit JSON under tests/fixtures/grading-honesty/. Regen via flag. | ✓ |
| Hand-authored stub responses | Hard-code stub-responses.sh per question. Brittle. | |
| Shared reusable stub library | Per-resource stub generators. Heavy scaffolding upfront. | |
| Re-use ref-solution.sh runs against same fixture | Test both empty=0 AND ref-solution=full from one fixture. | |

**User's choice:** Capture-then-replay
**Notes:** Adopted the spirit of option 4 too — a second `post-ref-solution-state.json` fixture is captured during the same operator step so the mocked test also guards the happy path. Captured in CONTEXT D-20.
