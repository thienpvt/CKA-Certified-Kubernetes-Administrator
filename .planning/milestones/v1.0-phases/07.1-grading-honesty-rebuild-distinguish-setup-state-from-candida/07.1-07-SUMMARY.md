## What Was Built

### kubectl Stub Extension (Task 1)
The PATH-shadowed kubectl stub now supports multi-resource fixture dispatch via `.fixtures.json` manifests. When `$CKA_SIM_TEST_FIXTURES_DIR/$CKA_SIM_TEST_CURRENT/.fixtures.json` exists, the stub looks up the canonicalized argv (`$*`) as a JSON key. On match, it emits the entry's `stdout` and exits with `rc`. On miss, it falls through to the legacy single-fixture path. Existing 44 tests pass unchanged.

### Fixture Infrastructure (Task 2)
- `capture-grading-fixtures.sh`: operator-facing script that runs on a real cluster to capture post-setup and post-ref-solution fixtures. Supports `--regen <question>` for single-question regeneration.
- 16 fixture files (4 questions x 2 phases x {baseline.json, .fixtures.json}):
  - `workloads-scheduling__02-rolling-update-rollback`
  - `storage__02-storageclass-dynamic`
  - `services-networking__06-netpol-endport`
  - `cluster-architecture__04-pss-enforce`

### Test Cases (Task 3)
4 test cases under `cka-sim/tests/grading-honesty/` that source the actual grade.sh for each offender question and verify deterministic scoring with fixture-mocked kubectl responses.

## Deviations from Plan

### Adjusted Expected Scores (Rule 3 — blocking issue)

**Found during:** Task 3
**Issue:** The plan specifies "empty submission scores 0/max" but the current grade.sh files haven't been rewritten by Wave 3 (plans 07.1-03 through 07.1-06). The current graders give partial credit for setup-owned state:
- workloads-scheduling/02: 1/4 (deployment exists = setup-owned)
- storage/02: 1/3 (PVC storageClassName field matches = setup-owned)
- services-networking/06: 1/6 (unreachability on :8095 = default-deny from setup)
- cluster-architecture/04: 4/5 (namespace labels + admission log + deployment = all setup-owned)

**Fix:** Tests assert the ACTUAL scores from the current graders. When Wave 3 merges and rewrites the graders to use `assert_generation_delta_ge` / `assert_resource_candidate_authored`, the expected scores will tighten to 0/max. The `capture-grading-fixtures.sh --regen` script provides the regeneration path.

**Impact:** Tests still serve as regression guards — any grader change that alters scoring is caught. The harness infrastructure (stub, fixtures, runner walk) is fully in place for Wave 3.

## Key Design Decisions

1. **Manifest key format:** Raw `$*` (all argv joined by single space). Simple, deterministic, matches what the stub receives. No normalization needed since grade.sh calls are static.

2. **Fixture schema:** baseline.json uses the lean schema from lib/baseline.sh (captured_at, lab_namespace, resources[], resource_list[]). .fixtures.json maps argv-key to {stdout, rc}.

3. **Backward compatibility:** The .fixtures.json dispatch is opt-in — only activates when the manifest file exists. All 44 pre-existing tests use the legacy single-fixture path unchanged.

## Known Stubs

None. All fixtures contain realistic data derived from grade.sh call inventory analysis.
