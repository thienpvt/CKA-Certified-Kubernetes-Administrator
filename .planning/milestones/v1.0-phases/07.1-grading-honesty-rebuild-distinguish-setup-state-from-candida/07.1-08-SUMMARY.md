## Deviations from Plan

### Rule 2 — Discovered canonical-kind lookup contract

**Found during:** Task 2 (test verification revealed Q03's `assert_changed_since_setup pv q03-retain-pv` passed when it should have failed for an unchanged resource).

**Issue:** `lib/baseline.sh::is_candidate_modified` derives the baseline lookup key as `lowercased-kind/name` from the caller's kind argument verbatim (no alias canonicalization). Meanwhile `lib/baseline.sh::capture` writes `resource_list` entries as `(.kind|ascii_downcase)/(.name)` from the real `kubectl get -o json` output, which always emits full kind names (`PersistentVolume`, not `PV`). Result: a grader calling `assert_changed_since_setup pv q03-retain-pv` looks up `pv/q03-retain-pv`, misses the baseline entry `persistentvolume/q03-retain-pv`, treats the resource as candidate-authored, and PASSES — defeating the whole purpose of the gate.

**Fix:** Updated `cka-sim/packs/storage/03-access-modes-reclaim/grade.sh` to call `assert_changed_since_setup persistentvolume q03-retain-pv` (canonical kind). Updated Q03 fixture .fixtures.json files to use the matching argv key (`get persistentvolume q03-retain-pv -o jsonpath=...`).

**Files modified:** `cka-sim/packs/storage/03-access-modes-reclaim/grade.sh`, `cka-sim/tests/fixtures/grading-honesty/storage__03-access-modes-reclaim/post-setup/.fixtures.json`, `cka-sim/tests/fixtures/grading-honesty/storage__03-access-modes-reclaim/post-ref-solution/.fixtures.json`.

**Commit:** 15337aa.

**Forward implication:** This is the first usage of `assert_changed_since_setup` across the codebase. Phase 07.1 Plan 13 (VERIFICATION) should document the canonical-kind contract for future callers; alternatively a future plan could update `is_candidate_modified` to canonicalize via `kubectl api-resources` lookup (out of scope here).
