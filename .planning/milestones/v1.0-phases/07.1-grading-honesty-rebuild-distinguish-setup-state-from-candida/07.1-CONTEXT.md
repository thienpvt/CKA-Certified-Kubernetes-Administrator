# Phase 07.1: Grading Honesty Rebuild — Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Make graders score candidate work, not setup state. Empty exam submissions of `blueprint-alpha` MUST score 0/100 across all 17 questions on a live cluster. The fix is structural: introduce a post-setup baseline mechanism, three new delta-aware assertion helpers, and a matching trap-detector ownership check, then audit and fix every grader in the 5 packs that currently awards points for state setup.sh already produced. Reference-solution round-trip (Phase 3 GRADE-06) must continue to pass for every question. CI gates the build on the new empty-submission contract via a mocked-kubectl per-question test. Phase exits green when (a) empty submission of `blueprint-alpha` returns 0/100 0 raw points 0 traps on live cluster, (b) CI fails any future PR that breaks that invariant, (c) ref-solution still PASSes everywhere, and (d) `GRADING-HONESTY.md` documents the baselining contract for question authors.

</domain>

<decisions>
## Implementation Decisions

### Baseline Capture Mechanism
- **D-01:** Baseline is **runner-managed**. The runner (`cka-sim drill`, `cka-sim exam`) calls a baseline-capture step AFTER `setup.sh` returns and BEFORE the candidate's prompt / before any grading. Setup-script authors do NOT call baseline logic themselves — this removes the "easy to forget" failure mode and lets the runner enforce the contract once.
- **D-02:** Baseline is written to `/tmp/cka-sim/<question-slug>/baseline.json` (per-question, outside the cluster, ephemeral). Path is exposed to graders + trap detectors via env var `CKA_SIM_BASELINE_PATH`. Directory is created with mode 0700 by the runner.
- **D-03:** Baseline schema is **lean**, per resource in the lab namespace:
  ```json
  {
    "captured_at": "<unix-epoch>",
    "lab_namespace": "cka-sim-<domain>-NN",
    "resources": [
      { "kind": "Deployment", "name": "web", "namespace": "cka-sim-...",
        "generation": 3, "resourceVersion": "12345",
        "labels": { "app": "web" } },
      ...
    ],
    "resource_list": [ "deployment/web", "pvc/data", ... ]
  }
  ```
  `resources[]` holds the per-resource fields for delta checks. `resource_list[]` is the flat `kind/name` set for authorship checks. ~1–5 KB per question. No content hashes, no full kubectl snapshots — keep payload small + jq-cheap.
- **D-04:** Capture scope is **every resource in the lab namespace** that matches a fixed kind allowlist (Deployment, StatefulSet, DaemonSet, Pod, Service, ConfigMap, Secret, PVC, PV, NetworkPolicy, RoleBinding, ClusterRoleBinding, ServiceAccount, plus per-question additions via env override `CKA_SIM_BASELINE_KINDS_EXTRA`). Cluster-scoped resources prefixed with the question ID are also captured. Planner finalizes the canonical allowlist.

### Delta-Helper API (lib/grade.sh)
- **D-05:** Three new helpers, all sourcing `CKA_SIM_BASELINE_PATH`:
  - `cka_sim::grade::assert_changed_since_setup <kind> <name> [-n <ns>] [<weight>]`
  - `cka_sim::grade::assert_generation_delta_ge <kind> <name> <N> [-n <ns>] [<weight>]`
  - `cka_sim::grade::assert_resource_candidate_authored <kind> <name> [-n <ns>] [<weight>]`
- **D-06:** `assert_changed_since_setup` passes iff current `.metadata.generation` > baseline.generation **OR** current `.metadata.resourceVersion` != baseline.resourceVersion. Generation covers spec edits; resourceVersion covers any kubectl-applied change (label, annotation). Cheap, robust, no spec diffing.
- **D-07:** `assert_generation_delta_ge N` passes iff (current.generation − baseline.generation) >= N. If the resource was not in the baseline (didn't exist), the helper FAILS (candidate did not edit an existing object — they should use `assert_resource_candidate_authored` instead).
- **D-08:** `assert_resource_candidate_authored` passes iff `<kind>/<name>` exists in current cluster state AND does NOT appear in `baseline.resource_list[]`. UID matching is not required for v1 — name+kind not-in-baseline is sufficient for the 4 offender cases.
- **D-09:** All three helpers share the existing accumulator contract (`CKA_SIM_GRADE_TOTAL` += weight, on pass append PASSES + ok, on fail append FAILS + err, never `die`). They look identical to the existing 7 assertion helpers from the caller's perspective.

### Trap-Detector Ownership (lib/traps.sh)
- **D-10:** Per-resource trap detectors (those that fire on the candidate's mutation of a specific resource, e.g., `default-sa-used` on a Deployment) are refactored to take an explicit subject (`<kind> <name> [-n <ns>]`) and call `assert_changed_since_setup` OR `assert_resource_candidate_authored` internally before firing. If the resource is unchanged-since-setup AND was created by setup, the detector returns empty — no trap recorded.
- **D-11:** Traps that legitimately fire on setup-state (e.g., content-bug traps like `kubelet-runtime-flag-in-kubeconfig`, `pss-error-string-mismatch`, `removed-container-runtime-flag` — they represent author-intentional state baked into the lab to be diagnosed) are tagged in `traps/catalog.yaml` with `ownership: setup-allowed` and skip the ownership gate. Default for everything else is `ownership: candidate-required`. Planner finalizes the per-trap tagging.
- **D-12:** The Q3 `default-sa-used` regression case is the canonical fix-test: detector must NOT fire on setup's nginx:1.25 Deployment when the candidate has done nothing.

### Per-Question Fixes (4 confirmed offenders)
- **D-13:** `workloads-scheduling/02-rolling-update-rollback` — assertion 4 swaps `generation >= 3` for `assert_generation_delta_ge web 2 -n $CKA_SIM_LAB_NS` (setup bumps gen to 3; candidate's rollout-forward + rollback must add at least 2 more). Assertion 3 (image == nginx:1.25) keeps absolute check but is gated on assertion 4 — only counted if candidate has actually rolled (otherwise the image being nginx:1.25 is meaningless setup-state).
- **D-14:** `storage/02-storageclass-dynamic` — assertion that asserts `pvc.spec.storageClassName == fast-ssd` is replaced with `assert_resource_candidate_authored storageclass fast-ssd` (candidate must create the StorageClass) + `assert_changed_since_setup pvc data -n $CKA_SIM_LAB_NS` (candidate must have edited the PVC).
- **D-15:** `services-networking/06-netpol-endport` — assertion 5 (client reaches :8085) is gated on `assert_resource_candidate_authored networkpolicy <name> -n $CKA_SIM_LAB_NS` — only score reachability if candidate has created a NetworkPolicy. Default-allow passing the reachability check no longer scores.
- **D-16:** `cluster-architecture/04-pss-enforce` — audit the one passing assertion; expected fix is gating on `assert_changed_since_setup namespace <ns>` for ns-label mutation OR `assert_resource_candidate_authored` for admission log artifact. Planner confirms during research.

### CI Regression Strategy
- **D-17:** CI enforces "empty submission scores 0" via a **mocked-kubectl unit test** added to `cka-sim/scripts/test.sh`. New test cases live under `cka-sim/tests/grading-honesty/<question>.sh`. Each test source's the question's grade.sh with the PATH-shadowed kubectl stub (Phase 2 mechanism) replaying a captured fixture, asserts the grader emits `SCORE: 0/<max>` and 0 traps. Runs on every PR via existing `bash-tests` GHA job — fast, no real cluster.
- **D-18:** Test fixtures are **capture-then-replay**: for each question, a one-time live-cluster capture step runs `setup.sh` against a real cluster and snapshots `kubectl get <kind> <name> -o json` for every resource grade.sh queries, plus the baseline.json the runner would produce. Snapshots commit to `cka-sim/tests/fixtures/grading-honesty/<question>/` (post-setup-state.json + baseline.json). Fixtures regenerate via a `--regen` flag on the capture helper script.
- **D-19:** Ref-solution round-trip preservation (success criterion 3) is verified by the existing live-cluster UAT discipline (Phase 3 GRADE-06), NOT by this mocked test. The mocked test only proves the empty-submission contract.
- **D-20:** A second fixture per question (`post-ref-solution-state.json`) is captured during the same one-time live-cluster step so the mocked test can ALSO verify the happy path (grade.sh against ref-solution state must score >0). Double-duty: empty=0 + ref=full both run on every PR.

### Plan Granularity
- **D-21:** Five-wave plan structure (~5 plans total):
  - **Wave 1 — Library + runner integration:** `lib/grade.sh` (3 new helpers), `lib/traps.sh` (ownership refactor + catalog.yaml `ownership:` tagging), runner integration (drill.sh + exam.sh baseline-capture hook + `CKA_SIM_BASELINE_PATH` export), `traps/catalog.yaml` schema bump.
  - **Wave 2 (parallel) — 4 confirmed offender fixes:** one plan per offender (D-13..D-16); each updates the question's grade.sh + captures fixtures + adds the per-question test case.
  - **Wave 3 (parallel) — Audit + fix remaining 13 questions:** grouped by pack into 2–3 plans; each plan runs empty-submission against its questions on live cluster, classifies failing assertions, applies the appropriate delta/authorship fix, captures fixtures.
  - **Wave 4 — CI harness + lint:** `cka-sim/scripts/test.sh` grading-honesty test suite wiring, CI workflow extension if needed.
  - **Wave 5 — Docs + verification:** new `cka-sim/GRADING-HONESTY.md` documenting the baselining contract for authors; `cka-sim/AUTHORING.md` cross-reference; Phase 7 README scoring-honesty caveat removal; 07.1-VERIFICATION.md.
- **D-22:** Audit escape — if a question's grading-honesty fix requires reworking the question's pedagogy (not a mechanical delta-check substitution), flag in 07.1-VERIFICATION.md, commit a temporary skip in the CI test for that question (with TODO + follow-up tracking), ship 07.1 with the other 16. Avoid stalling the whole phase on one tricky question.

### Claude's Discretion
- Exact baseline kind allowlist (D-04) — start with the 14 listed kinds; planner adjusts during research.
- Per-trap `ownership:` tagging in `traps/catalog.yaml` (D-11) — planner enumerates during Wave 1 research.
- Whether Wave 3 audit splits by pack (5 plans) or by classification (setup-collision / default-allow / trap-on-setup) — pick what minimizes context-switching between fixtures.
- Bash structure of the runner baseline-capture hook (D-01) — likely a new `cka-sim/lib/baseline.sh` with `cka_sim::baseline::capture <ns>` called by drill.sh after setup, by exam.sh per-question after setup. Planner picks helper boundaries.
- Capture-helper script location (D-18) — likely `cka-sim/scripts/capture-grading-fixtures.sh` with `--regen [<question>]` flag.

### Established Patterns

<specifics>
## Specific Ideas

- Baseline.json schema is the load-bearing contract — keep it small + flat + jq-friendly. No nested resource trees. One array of resources, one flat resource_list.
- Capture-helper script (`cka-sim/scripts/capture-grading-fixtures.sh`) is operator-facing, not CI-facing: maintainer runs it once on a real cluster to seed `tests/fixtures/grading-honesty/`. `--regen <question>` regenerates a single question's fixtures after grader/setup edits.
- Wave 2 fixes treat the four confirmed questions as the canonical reference patterns. Whatever shape Wave 2 lands becomes the playbook Wave 3 follows.
- Test naming convention: `tests/grading-honesty/<pack>__<slug>.sh` (double-underscore separates pack + slug). Each test case sources fixtures from `tests/fixtures/grading-honesty/<pack>/<slug>/`.
- README caveat removal (Wave 5) is a single edit in `cka-sim/README.md` once CI is green; gate with `git grep -l 'scoring-honesty caveat'` so it's mechanical.

</specifics>
