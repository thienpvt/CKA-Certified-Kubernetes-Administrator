# Phase 07.1: Grading Honesty Rebuild — Cross-AI Reviews

## Reviewer: Codex (OpenAI)

### Summary

Plans cover correct core idea: runner-managed baseline, delta-aware helpers, trap ownership, offender fixes, and CI regression. Main risk: scope too large and fixture-heavy. Plans 08-12 try to audit 30 questions plus hand-author many mocked fixtures. That can become brittle and may create false confidence, especially for file-edit, SSH, node-action, rollout, exec, and wait behavior. Biggest design gap: baseline schema and helper semantics are not fully aligned with cluster-scoped resources, namespaces, resourceVersion noise, and exam-mode per-question baseline path during batch grading.

### Strengths

- Runner-managed baseline is right. Avoids relying on question authors.
- Helper split is good:
  - existing resource modified: `assert_changed_since_setup`
  - multiple spec changes: `assert_generation_delta_ge`
  - new resource: `assert_resource_candidate_authored`
- Trap ownership gate addresses real false-positive class.
- Four known offenders have concrete fixes.
- CI regression intent is correct: empty submission must score `0/N`.
- Documentation plan is solid and includes authoring contract.
- Audit-escape concept is pragmatic for file/node tasks outside current baseline schema.

### Concerns

| # | Severity | Concern |
|---|----------|---------|
| 1 | HIGH | **Fixture scope explosion.** Plans 08-12 require ~120+ fixture files plus many test scripts. Hand-authored `.fixtures.json` can drift from real `kubectl` behavior and pass while live graders fail. |
| 2 | HIGH | **Audit-escape weakens phase success.** Roadmap success says empty `blueprint-alpha` scores `0/100`, ref-solution passes. Plans allow `SCORE: 0/0` or thin scoring for file/node tasks. That may satisfy empty-score but not grading honesty for real candidate work. |
| 3 | HIGH | **Exam mode baseline path may break batch grading.** `CKA_SIM_BASELINE_PATH` is one env var. If exam grades questions later, `grade.sh` must get each question's own baseline path before each grade call. Plan 02 only describes export during setup. |
| 4 | HIGH | **`resourceVersion` noise can false-pass.** `assert_changed_since_setup` using rv can pass from controller/status churn. Sleep helps but does not eliminate it, especially Pods, PVCs, Deployments. |
| 5 | MEDIUM | **Cluster-scoped baseline capture underspecified.** D-04 includes PV, StorageClass, ClusterRoleBinding, etc., but Plan 01 capture command only captures namespaced allowlist. `assert_resource_candidate_authored storageclass fast-ssd` needs baseline coverage for cluster-scoped kinds. |
| 6 | MEDIUM | **Kind token normalization risk.** `kubectl get deployment -o name` outputs `deployment.apps/web`; baseline key uses `deployment/web`. Helpers must normalize consistently or only check existence via rc. |
| 7 | MEDIUM | **`weight=0` helper behavior may still append pass/fail arrays and affect output.** Fine for messages, but tests parsing `SCORE:` only may miss noisy misleading passes. |
| 8 | MEDIUM | **`capture` graceful-degrade to empty baseline is dangerous.** If capture fails, candidate-authored checks may treat all existing setup resources as candidate-authored. Better fail hard in runner or mark grading unavailable. |
| 9 | MEDIUM | **`is_candidate_modified` back-compat returns 0 on missing baseline.** Good for old tests, risky in real runner if baseline missing. Need mode distinction: back-compat for unit tests, strict for runtime. |
| 10 | MEDIUM | **Trap catalog `ownership` field not actually used by detector logic in Plan 02.** Plan adds tags but gates detectors directly. Either catalog drives behavior or tags are documentation only. |
| 11 | LOW | **TDD "RED then commit" is process-heavy.** Fine for agents, but not necessary to preserve in final code history unless workflow requires commits. |
| 12 | LOW | **README caveat removal before live UAT may be premature.** Remove only after mocked and live empty exam pass. |

### Suggestions

1. **Add strict runtime mode:** In runner: if baseline capture fails, abort question setup or mark question ungradable. Keep back-compat only under tests via `CKA_SIM_BASELINE_BACKCOMPAT=1`.
2. **Fix exam batch grading contract:** Store baseline path per question in exam state. Before each `grade.sh`, export that question's `CKA_SIM_BASELINE_PATH`.
3. **Add cluster-scoped capture in Plan 01:** StorageClass, PV, ClusterRoleBinding, ClusterRole, PriorityClass, CRD.
4. **Reduce Wave 5 scope:** Prioritize 17 `blueprint-alpha` questions first. Defer non-blueprint pack questions to follow-up.
5. **Prefer generated fixtures over hand-authored fixtures:** Make `capture-grading-fixtures.sh` produce `.fixtures.json` from trace logs.
6. **Add one live-cluster smoke before caveat removal:** Empty `blueprint-alpha` exam + one ref-solution round trip for at least the four offenders.
7. **Make `resourceVersion` checks opt-in or safer:** Use generation where possible. For ConfigMap/Secret/Namespace, rv is unavoidable, but document noise risk.
8. **For audit-escape questions, do not use `SCORE: 0/0` as success.** Mark unsupported in CI and require live UAT. Or add file baseline now.

### Risk Assessment

**Overall risk: MEDIUM-HIGH.**

Core architecture is sound, but execution plan is large and fixture-heavy. Highest risk sits in CI giving false confidence because many graders depend on live cluster behavior, node files, SSH, rollout history, `kubectl exec`, and status timing. Phase can achieve "empty scores 0" for API-resource questions, but full phase goal needs stricter baseline failure handling, exam batch baseline mapping, cluster-scoped capture, and fewer audit-escape gaps.

## Action Items for Replanning

The following HIGH-severity concerns should be addressed before execution:

1. **Concern #3 (exam batch baseline path):** Plan 02 must explicitly describe how exam.sh exports per-question `CKA_SIM_BASELINE_PATH` before each batch-grade call. The path is already per-question (`/tmp/cka-sim/<slug>/baseline.json`) — the plan just needs to document the exam.sh batch-grading loop re-exporting it.

2. **Concern #5 (cluster-scoped capture):** Plan 01's capture function must include a second `kubectl get` for cluster-scoped kinds (StorageClass, PV, ClusterRoleBinding, PriorityClass) and merge into baseline.json. RESEARCH Q2 already specifies this — plan task needs to be explicit.

3. **Concern #6 (kind normalization):** Plan 01 must specify the canonical token format (`lowercase-kind/name`, NOT `kind.apigroup/name`). The capture script must normalize `kubectl get -o name` output (strip `.apps`, `.rbac.authorization.k8s.io`, etc.) before writing `resource_list[]`.

4. **Concern #4 (rv noise):** Document in GRADING-HONESTY.md that `assert_changed_since_setup` should prefer generation where available; rv is the fallback for kinds without generation (ConfigMap, Secret, Namespace labels). The 1-2s sleep after setup absorbs most controller flap but is not a guarantee.

5. **Concern #8 (capture failure):** Plan 01 should specify: if `cka_sim::baseline::capture` fails (jq error, kubectl unreachable), the runner aborts the question with a clear error rather than proceeding with no baseline.

**Non-blocking (address during execution or follow-up):**
- Concern #1 (fixture explosion): mitigated by `capture-grading-fixtures.sh --regen` generating fixtures from live cluster, not hand-authoring.
- Concern #2 (audit-escape): accepted per CONTEXT D-22 — flag in VERIFICATION.md, don't block phase.
- Concern #12 (README caveat): move removal to AFTER live UAT confirms 0/100 (already the plan — Wave 6 is last).
