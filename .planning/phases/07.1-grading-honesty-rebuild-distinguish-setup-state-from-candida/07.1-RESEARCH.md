# Phase 07.1: Grading Honesty Rebuild — RESEARCH

## Q1: `.metadata.generation` and `.metadata.resourceVersion` semantics

### What apiserver actually does
- **`.metadata.generation`** — incremented ONLY when `.spec` changes. Status-only updates and metadata-only updates (label/annotation edits) do NOT bump it. The kubelet/controllers' status writes are status-subresource updates → no generation bump.
- **`.metadata.resourceVersion`** — incremented on EVERY update including status writes, controller heartbeats, condition transitions. A reliable "something happened" signal but noisy: a background controller writing `.status.readyReplicas` or `.status.observedGeneration` after setup completes will bump rv without candidate involvement.

### Per-kind reliability for `assert_changed_since_setup`

| Kind | generation bumps on spec edit? | resourceVersion bumps on status flap? | Practical signal |
|------|-------------------------------|---------------------------------------|------------------|
| Deployment, StatefulSet, DaemonSet | ✓ | ✓ frequently (replica/condition transitions) | gen-OR-rv: gen catches spec edits, rv catches label/annotation |
| Pod | usually frozen at 1 (immutable spec) | ✓ very frequently | Pod gen is unreliable; rely on rv only |
| Service | ✓ | rare (Service has no controller flap) | gen-OR-rv works well |
| ConfigMap, Secret | NOT bumped (no .spec) — generation stays at 1 | only on edit | rv-only signal |
| PVC | ✓ but most fields immutable post-Bind | binder writes phase via status → rv flaps | gen-OR-rv works; binder writes happen during setup, before baseline capture |
| StorageClass | ✓ | rare | gen-OR-rv works |
| NetworkPolicy | ✓ | rare | gen-OR-rv works; cluster-architecture/04 case |
| Namespace | ✓ on label/annotation edits via spec? — actually labels are metadata so generation does NOT bump on `kubectl label namespace`. rv bumps. | ✓ on phase | rv-only for label edits; the cluster-architecture/04 case |

### CONTEXT D-06 verdict (gen OR rv)
The OR is correct. Confirmed false-positive risk: a controller status flap between setup-end and baseline-capture would bump rv. **Mitigation: runner captures baseline AFTER setup.sh returns 0** (per D-01). Setup scripts already include `kubectl wait --for=condition=Available` calls that block until the controller has settled, so the post-wait state is stable. Risk is bounded but not zero — for noisy controllers (Deployments mid-rollout) the runner should sleep briefly (~1–2 s) after setup returns before capture.

### Refinement to D-06
For Namespace label edits (cluster-architecture/04 candidate work pattern), rely on `resourceVersion` only — `kubectl label namespace` does not bump generation. Document this in `GRADING-HONESTY.md` so authors don't author `assert_generation_delta_ge namespace ...` and watch it always fail.

## Q2: jq patterns for the lean baseline.json

### Schema (locked by D-03)
```json
{
  "captured_at": 1747000000,
  "lab_namespace": "cka-sim-storage-02",
  "resources": [
    { "kind": "Deployment", "name": "web",
      "namespace": "cka-sim-workloads-scheduling-02",
      "generation": 3, "resourceVersion": "12345",
      "labels": { "app": "web" } }
  ],
  "resource_list": [ "deployment/web", "pvc/data" ]
}
```

### Idiomatic jq one-liners

**Membership in resource_list (for `assert_resource_candidate_authored`):**
```bash
# Returns "true" if candidate authored (kind/name not in baseline)
in_baseline=$(jq -r --arg key "deployment/web" \
  '(.resource_list // []) | index($key) != null' "$CKA_SIM_BASELINE_PATH")
```
- `deployment/web` is the canonical lookup token. lower-cased kind + slash + name. Cluster-scoped resources omit namespace (e.g., `storageclass/fast-ssd`); namespaced resources also use `kind/name` since each baseline.json belongs to ONE lab namespace.

**Per-resource lookup (for `assert_changed_since_setup` and `assert_generation_delta_ge`):**
```bash
baseline_gen=$(jq -r --arg k "Deployment" --arg n "web" \
  '(.resources[] | select(.kind==$k and .name==$n) | .generation) // empty' \
  "$CKA_SIM_BASELINE_PATH")

baseline_rv=$(jq -r --arg k "Deployment" --arg n "web" \
  '(.resources[] | select(.kind==$k and .name==$n) | .resourceVersion) // empty' \
  "$CKA_SIM_BASELINE_PATH")
```
- `// empty` returns blank when the resource was not in baseline → caller can detect "candidate-authored" without needing a separate check, but `assert_resource_candidate_authored` keys off `resource_list` for clarity.

**Bash compare (caller side):**
```bash
current_gen=$(kubectl get deployment web -n "$ns" -o jsonpath='{.metadata.generation}' 2>/dev/null)
if [[ -z "$baseline_gen" ]]; then
  # Resource not in baseline — for assert_changed_since_setup this should fail
  # (caller wanted a delta on something that should have existed at setup)
  return 1
fi
(( current_gen > baseline_gen )) || ...
```

### Capture-side jq (runner)

`cka_sim::baseline::capture <ns>` produces the JSON via:
```bash
kubectl get -n "$ns" \
  deployment,statefulset,daemonset,pod,service,configmap,secret,pvc,networkpolicy,rolebinding,serviceaccount \
  -o json 2>/dev/null \
| jq '{
    captured_at: now | floor,
    lab_namespace: $ns,
    resources: [ .items[] | {
      kind, name: .metadata.name, namespace: .metadata.namespace,
      generation: (.metadata.generation // null),
      resourceVersion: .metadata.resourceVersion,
      labels: (.metadata.labels // {})
    } ],
    resource_list: [ .items[] | (.kind | ascii_downcase) + "/" + .metadata.name ]
  }' --arg ns "$ns"
```
- Cluster-scoped resources (StorageClass, PV, ClusterRoleBinding) require a separate `kubectl get` since they ignore `-n` — runner runs two queries and merges via jq `+`.
- Resources not in the namespace (kind allowlist returned 0 items) just contribute empty arrays — jq handles this without special-casing.

## Q3: PATH-shadowed kubectl stub — what it can and can't do

### Current capability (`cka-sim/tests/bin/kubectl`)
- Reads ONE env var pair: `CKA_SIM_TEST_FIXTURES_DIR` (root) + `CKA_SIM_TEST_CURRENT` (sub-path) → fixture file = `${root}/${current}.json`.
- Supports verbs: `get`, `auth can-i`, `exec`, `describe`. Translates `-o jsonpath={.foo.bar}` → jq via sed.
- Returns exit 1 on missing fixture, exit 64 on unsupported verb.

### Limitation for grading-honesty tests
**Each `CKA_SIM_TEST_CURRENT` points at a SINGLE fixture file.** Per-question grading-honesty tests need to query MULTIPLE resources during one grade.sh run (e.g., `kubectl get deployment web` AND `kubectl get pvc data` AND `kubectl get networkpolicy ...`). The current stub returns the same JSON for every call — won't work.

### Recommended extension (Wave 4 task)
Extend the stub to support per-resource fixtures by routing argv through a path resolver:

```bash
# When -o name or no -o flag: look up fixture by kind+name+ns
# Path convention: ${CKA_SIM_TEST_CURRENT}/<kind>-<name>[__-n_<ns>].json
# Example: tests/fixtures/grading-honesty/storage__02-storageclass-dynamic/post-setup/
#           pvc-app-cache.json
#           storageclass-fast-ssd.json (404 → exit 1 = NotFound)
```

Or simpler: extend stub to read a manifest at `${CKA_SIM_TEST_CURRENT}/.fixtures.json` mapping argv-fingerprints to fixture files. This keeps the stub pure-bash without per-resource file proliferation.

**Pitfall captured:** the stub's jsonpath translator handles `{.foo}`, `[?(@.x=="y")]`, and `[*]` but not nested filters. The 3 new helpers' kubectl calls (`kubectl get <kind> <name> -o jsonpath='{.metadata.generation}'`) all use simple paths — within the translator's support range.

### Capture pitfalls (one-time live cluster step)
- `kubectl get -o json` includes `managedFields`, which bloats fixtures by 5–10x. Capture script should strip it: `| jq 'del(.metadata.managedFields)'` before write.
- `creationTimestamp`, `resourceVersion`, `uid` change every cluster run — fixtures freeze them as snapshots. The grading-honesty test compares against baseline RV, NOT against a hard-coded value, so this is fine; just commit the fixtures as captured.
- `.status.observedGeneration` and `.status.conditions[].lastTransitionTime` are noisy. Strip during capture: `| jq 'walk(if type=="object" and has("lastTransitionTime") then del(.lastTransitionTime) else . end)'` to keep diffs minimal.

## Q4: 4 confirmed offenders — exact leaks + exact fixes

### workloads-scheduling/02-rolling-update-rollback (UAT: 4/4 free)

**Current grade.sh assertions (4):**
1. `assert_resource_exists deployment web -n $LAB_NS` → setup creates it. **Leaks 1.**
2. `kubectl rollout status deployment/web --timeout=60s` → setup waited for rollout already. **Leaks 1.**
3. `assert_field_eq deployment web '{.spec.template.spec.containers[0].image}' 'nginx:1.25' -n $LAB_NS` → setup creates with this image. **Leaks 1.**
4. Inline check: `gen=$(kubectl get deployment web -o jsonpath='{.metadata.generation}'); [[ gen >= 3 ]]` → setup bumps gen to 3 via annotate + strategic-merge patch. **Leaks 1.**

**Total leak: 4/4 = 100% — empty submission scores 4/4.**

**Fixes:**
- Assertion 1 (resource_exists): keep — every grader has a "the resource is present" sanity check; setup-state collision here is acceptable IF a downstream delta assertion gates the score. **Mitigation: only count assertion 1's point when assertion 4 passes. Implementation: weight=0 unless companion assertion fired.** Or simpler — drop assertion 1's weight to 0 and treat it purely as a precondition.
- Assertion 2 (rollout status): same — setup-state. Drop weight to 0; treat as precondition.
- Assertion 3 (image == nginx:1.25): structurally tautological — the question's GOAL is to roll forward THEN rollback to nginx:1.25, so final state == initial state. Cannot detect from final state alone. **Two options:**
  - (a) Replace with a check on `kubectl rollout history` count (≥ 3 entries).
  - (b) Drop this assertion; rely on assertion 4's delta to prove rollout activity happened.
  - **Recommended (b)** — simpler, matches CONTEXT D-13.
- Assertion 4 (generation >= 3): replace with `cka_sim::grade::assert_generation_delta_ge deployment web 2 -n $LAB_NS`. Setup leaves gen at 3; candidate's rollout-forward bumps to 4, rollback to 5 → delta = 2. The `>= 2` threshold passes only if BOTH a forward AND a backward rollout actually happened.

**Result:** Empty submission → assertions 1+2 weight 0 (or removed), 3 removed, 4 fails → score 0/1 (just the delta assertion).

## Q6: `lib/traps.sh` refactor pitfalls

### Current detector signature
`cka_sim::trap::detect_<id> [<args>]` → stdout = trap-id on hit, EMPTY on miss. No global state. No side effects.

### Per-resource detectors (subject to ownership gate)
- `detect_default_sa_used <ns> <pod>` — currently fires on setup's pods (workloads-scheduling/02 case)
- `detect_missing_dns_egress <ns> <netpol>` — fires on candidate-authored NP only? Need to confirm
- `detect_hostpath_pv_without_nodeaffinity <pv>` — PV is cluster-scoped; may be candidate-authored
- `detect_rbac_viewer_role_mismatch <ns> <role>` — Role likely candidate-authored
- `detect_service_label_mismatch <ns> <service>` — Service may be either

### Text-based detectors (no ownership gate needed)
- `detect_pss_error_string_mismatch <text>` — reads candidate file; ALWAYS candidate's text
- `detect_psp_fictional_pod_label_exemption <text>` — same
- `detect_kubelet_runtime_flag_in_kubeconfig <text>` — same
- `detect_removed_container_runtime_flag <text>` — same
- `detect_as_flag_format_wrong <text>` — same

### Recommended refactor pattern
Add ONE helper to `lib/baseline.sh`:
```bash
# cka_sim::baseline::is_candidate_modified <kind> <name> [-n <ns>]
# Returns 0 (true) if:
#   (a) baseline.json is missing/unset → BACK-COMPAT, fire freely (drill mode without baseline)
#   (b) resource not in baseline.resource_list (candidate-authored)
#   (c) current generation > baseline.generation OR current rv != baseline.rv
# Returns 1 (false) if resource exists in baseline AND is unchanged.
cka_sim::baseline::is_candidate_modified() { ... }
```

Per-resource detectors prepend ONE call before their existing logic:
```bash
cka_sim::trap::detect_default_sa_used() {
  local ns="$1" pod="$2"
  cka_sim::baseline::is_candidate_modified pod "$pod" -n "$ns" || return 0
  # ...existing logic unchanged...
}
```

**Back-compat:** if `CKA_SIM_BASELINE_PATH` is unset OR file missing, helper returns 0 (assume modified) → all existing tests continue to pass without baseline.json. Tests that verify ownership gating set `CKA_SIM_BASELINE_PATH` to a fixture. **This is the critical pitfall to avoid:** a strict-mode helper would break every existing test.

### Catalog `ownership:` field
Skip the gate for detectors tagged `ownership: setup-allowed` in catalog.yaml. Recommendation: add the field as optional with `candidate-required` default. Detectors don't read the field directly — `cka_sim::trap::should_check_ownership <id>` reads it lazily via the existing catalog parser. Only ~5 of the 11 detectors need `setup-allowed` (the text-based content-bug traps).

### Bash function signature pitfalls
- `cka_sim::baseline::is_candidate_modified` MUST handle `-n <ns>` flag parsing (matches existing `assert_*` helper convention). Reuse the same flag-parser pattern from `assert_resource_exists`.
- The helper must NOT call `die` on missing baseline — `return 0` (back-compat). Other failures (jq parse error, malformed baseline.json) MUST `die` (developer bug).

## Q7: Baseline lifecycle

### Lifecycle (one drill cycle)
1. Runner `cka_sim::drill::main` → `bash reset.sh` (idempotent cleanup, including any old `/tmp/cka-sim/<question>/baseline.json`).
2. Runner → `bash setup.sh` (creates lab ns + seed resources; existing setup helpers `wait_for_ns_active` block until controllers settle).
3. **NEW:** Runner → `cka_sim::baseline::capture "$CKA_SIM_LAB_NS"` (writes `/tmp/cka-sim/<question-slug>/baseline.json`, exports `CKA_SIM_BASELINE_PATH`). 1–2 s sleep BEFORE capture absorbs any final controller status flap.
4. Runner → prompt → candidate works on cluster.
5. Runner → `bash grade.sh` (reads baseline path via env var; new helpers + refactored detectors gate on baseline).
6. Runner → EXIT trap → `bash reset.sh` (deletes lab ns; should also `rm -f /tmp/cka-sim/<question-slug>/baseline.json` to avoid stale baseline poisoning a re-run).

### TRIP-02 idempotency (drill twice)
- `reset.sh` run 1 (start of drill 1): no baseline yet, idempotent.
- `setup.sh` run 1 → baseline captured at end of step 3.
- Drill 1 grade + EXIT-trap reset → baseline.json deleted.
- Second `cka-sim drill <pack>` invocation → repeats cycle from a clean slate.

### Exam mode (Phase 7)
- Each question gets its own per-question baseline path: `/tmp/cka-sim/<question-slug>/baseline.json`.
- Exam runner per-question: setup → capture → present → (advance / flag / skip / pause) → at end-of-exam batch grading reads each per-question baseline.
- Pause/resume safe: baseline is on disk, survives Ctrl-Z + fg. The `/tmp/cka-sim/` dir is recreated by setup if missing on resume.

### Edge: candidate accidentally writes to `/tmp/cka-sim/<question>/baseline.json`
- The path is in `/tmp/cka-sim/` not `/tmp/` directly. Reasonable directory hygiene. Document in `GRADING-HONESTY.md` that this dir is reserved.
- Optional defense: write baseline.json with mode 0444 (read-only) so accidental edits fail. Recommended.

### Reset.sh extension
Every `reset.sh` MUST `rm -rf /tmp/cka-sim/$question-slug/` to clean up baseline + any sandbox files (cluster-architecture/04 already has `/tmp/q04-pss-enforce/`). Audit during Wave 2 to verify each pack's reset.sh follows this. Add a lint check (in `lint-packs.sh`) that every reset.sh contains a `rm -rf /tmp/cka-sim/...` line referencing its question-slug.
